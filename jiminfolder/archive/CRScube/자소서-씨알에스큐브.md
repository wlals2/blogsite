## 질문 1. 회사에 지원이유가 무엇인가요? 지원 동기에 대해 말씀 부탁드립니다.

온프레미스 Kubernetes 클러스터를 직접 구축하고 220일간 운영하면서 가장 크게 배운 것은, "장애는 무조건 온다, 문제는 얼마나 빨리 감지하고 복구하느냐"라는 것이었습니다. kubelet이 중단되어 Pod이 Terminating 상태로 멈추고, FD가 고갈되어 Falco가 CrashLoopBackOff에 빠지는 상황을 직접 겪으면서, 장애를 막는 것보다 장애에 빠르게 대응하는 체계를 만드는 것이 더 중요하다는 것을 몸으로 익혔습니다.

그래서 장애가 터진 후가 아닌, 이상 징후 단계에서 선제적으로 감지하고 대응하는 체계를 직접 설계했습니다. 이 경험을 하면서 자연스럽게 "이런 역량이 가장 필요한 곳은 어디일까"를 생각하게 되었고, SRE 직무를 찾아보는 과정에서 CRScube를 알게 되었습니다.

임상시험은 일반 서비스와 다릅니다. 쇼핑몰이 10분 멈추면 매출 손실이지만, 임상시험 시스템이 멈추면 연구 데이터의 무결성이 깨지고 환자 안전에 직결됩니다. FDA, GxP 등 규제 환경에서 99.99% 가용성을 유지해야 한다는 것은, 인프라 안정성의 기준 자체가 다른 곳보다 높다는 뜻입니다.

220일간 장애 선제 대응 체계를 직접 만들고 운영해온 경험을 갖고 있으며, 이 역량이 가장 필요한 곳이 CRScube입니다. cubeCDMS·cubeSAFETY 등 10개 솔루션이 20개국에서 99.99% 가용성을 유지하는 데 실질적으로 기여하는 SRE가 되겠습니다.

## 질문 2. 본인이 지원 직무에 적합하다고 생각하시는 이유가 무엇인가요?

### 1. 컨테이너 기반 배포 자동화 경험

EKS(관리형)와 베어메탈 kubeadm(직접 구축) 양쪽을 운영하며 컨테이너 네트워킹, 스토리지, 배포 전략, 롤백 메커니즘의 원리를 깊이 이해하고 있습니다.

온프레미스 K8s 운영 시, 수동 배포의 위험성을 고려하여 GitOps 정책(수동 변경 금지)을 수립하고 ArgoCD를 도입했습니다. Canary 배포(Argo Rollouts)를 적용하여 신규 버전의 트래픽을 단계적으로 전환하는 무중단 배포 파이프라인을 구현했고, 501회 배포 실패 0, 문제 발생 시 롤백 10초의 결과를 냈습니다. 컨테이너 오케스트레이션의 핵심 원리는 동일하며, ECS 환경에도 빠르게 적응하겠습니다.

### 2. Terraform 기반 IaC 및 DR 설계 경험

멀티 클라우드 DR 프로젝트(팀장/4명)에서 Terraform S3 Backend + DynamoDB Lock으로 협업 가능한 IaC 환경을 구성했습니다. Cold Standby와 Warm Standby의 트레이드오프를 분석하고, Cold Standby 아키텍처로 비용 98% 절감과 RTO 28분을 동시에 달성했습니다. 씨알에스큐브가 북미·일본·중국으로 글로벌 확장하는 과정에서, 이 경험을 실무에 바로 적용할 수 있습니다.

### 3. 모니터링 및 관측성 체계 구축 경험

온프레미스 K8s 프로젝트에서 Prometheus + Grafana + Loki 기반 모니터링 체계를 구축하고, OpenTelemetry + Tempo를 활용한 분산 트레이싱도 직접 구축·운영했습니다. 실제로 분산 트레이싱을 통해 코드를 보지 않고도 외부 API의 1.5초 지연과 N+1 쿼리 병목을 정확히 짚어냈고, SLO 기반 알람 설계로 Discord 알람 오탐률 0%를 유지하고 있습니다. 도구는 Pinpoint·ELK와 다르지만 관측성의 원리와 설계 방식은 동일하며, AWS 환경에서도 빠르게 적응하겠습니다.

### 4. 보안 관제 및 네트워크 실무 경험

넷코아텍에서 11개월간 DMZ 구간 내 UTM 기반 경계 보안 아키텍처를 설계·구축한 경험이 있습니다. 방화벽 정책 최적화, 보안 존 세분화, IPS/IDS/Anti-Virus 맞춤형 정책 적용 등 네트워크 보안 실무를 수행했습니다. 또한 온프레미스 K8s에서 Cilium(L3/L4) + Istio(L7) 계층별 네트워크 정책을 설계하고, Falco + Wazuh를 결합한 보안 관제 체계도 직접 구축했습니다. 이 체계를 통해 SSH Brute Force, Directory Scanning 등 실제 외부 공격을 탐지하고 대응한 경험이 있으며, 임상시험이라는 규제 환경에서 요구하는 보안 수준에 대응하는 데 이 실무 경험이 기반이 됩니다.

### 5. 직접 부딪히며 익히는 실행력

쿠버네티스를 처음 공부할 때, PC 장비에 직접 클러스터를 구축하면서 FD 고갈, Cilium의 kube-proxy 대체 문제 등 수많은 장애를 겪었고, 그 과정에서 리눅스 커널과 네트워크 설정을 몸으로 익혔습니다. 현재 220일간 무중단 운영을 이어가고 있으며, 겪은 장애와 해결 과정을 기술 블로그(blog.jiminhome.shop)에 기록하고 있습니다.

멀티 클라우드 프로젝트 당시에는 Karpenter 운영 문제로 한계에 부딪혔을 때, AWS KRUG(한국 사용자 모임)에 참여해 현직 엔지니어 분들께 조언을 구하고, 알게 된 노하우를 동료들에게 전파하여 표창장을 받았습니다. 새로운 환경에서도 이처럼 직접 부딪히고 빠르게 익혀온 방식으로 적응하겠습니다.

장애를 겪고 끝내는 것이 아니라, 같은 장애가 반복되지 않도록 체계를 만드는 것이 제가 일하는 방식입니다. 배포 자동화부터 DR 설계, 관측성, 보안 관제까지 이 방식으로 직접 구축하고 운영해왔으며, CRScube의 글로벌 서비스 안정성에 기여하겠습니다.
