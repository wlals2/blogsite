## 2. 제품 포트폴리오

### 2-1. AGATHA (아가타) — Live

**핵심**: 공격 인프라 **사전** 탐지 CTI 서비스

```
기존 CTI: 공격 발생 → IOC 수집 → 분석 → "이미 당한 후"
AGATHA:   공격 준비 단계 → 인프라 포착 → "공격 전에 알 수 있다"
```

**MITRE ATT&CK 매핑**:
- 기존 CTI → Initial Access ~ Impact (공격 실행 단계)
- AGATHA → **Reconnaissance + Resource Development** (공격 준비 단계, PRE-ATT&CK)

**추적 대상**:
1. 국가배후 해킹조직의 **공격도구 프로파일링** + 인프라 정보
2. **피싱/스미싱** 인프라
3. **Simbox** (발신번호 변작장비) 인프라
4. **VPN/Proxy/Tor** 악성 IP 정보
5. 해커 그룹 프로파일링 + 국가별 인프라 정보

**기술적 차별점**:
- 악성코드 파일을 확보/분석하지 않아도 공격 준비 단계의 인프라를 포착
- C2 서버의 fingerprint (인증서 패턴, 서버 응답 특성 등)로 프로파일링
- 아직 무장화(weaponization)되지 않은 인프라도 추적 가능

**실전 검증 (언론 보도)**:
- 2024년 3월: 북한 해킹그룹 **안다리엘(Andariel)**이 국내 대기업 타깃 공격 정황 사전 포착
  - 경유지 서버(스웨덴 IP)와 안다리엘 악성 도메인 일치 확인
  - 유관기관에 위협정보 즉각 전달
- 2024~2025년: 북한 **킴수키(Kimsuky)**의 애플 아이클라우드 공격 포착
  - 세션 정보 탈취 → ID/PW 없이 아이클라우드 접근
  - 가짜 애플 도메인 사용, 피해 사용자별 폴더로 탈취 데이터 관리

### 2-2. ARTHUR (아서) — Beta (2025년 9월 출시)

**핵심**: 다크웹 위협 인텔리전스 서비스

```
기존 다크웹 CTI: .onion 히든 서비스 내용만 수집
ARTHUR:          .onion 히든 서비스의 실제 Public IP를 식별
```

**핵심 기능**:
- **다크웹 히든 서비스(.onion)의 Public IP 직접 식별** ← 이것이 핵심 차별점
- 100여 종 이상의 다크웹 히든 서비스 스캔
- SSH, SMTP 등 100여 개 네트워크 서비스 스캔
- 랜섬웨어 유출 사이트, 다크웹 불법 포럼, 텔레그램 데이터 검색
- 사용자 정의 키워드 등록 + 실시간 알림

**왜 중요한가**:
- Tor 히든 서비스는 익명성이 핵심 → IP를 알 수 없음이 기본
- ARTHUR가 이 IP를 식별할 수 있다면 → 랜섬웨어 운영자, 데이터 거래 포럼의 실제 서버 위치 파악 가능
- 수사기관, CERT, 기업 보안팀에게 핵심 인텔리전스

### 2-3. DASHIELL (대시엘) — Coming Soon

**추정**: ASM (Attack Surface Management) = 공격 표면 인텔리전스

- 아직 공개 정보 없음
- ASM은 "우리 조직의 외부 노출 자산을 자동으로 발견하고 취약점을 식별"하는 서비스
- AGATHA(공격자 인프라 탐지) + ARTHUR(다크웹) + DASHIELL(방어자 자산 파악) → 풀 스펙트럼 CTI

## 3. 기술 스택 (JD + 추론)

### JD에서 명시된 것
```
- Linux OS
- Shell, Python
- NoSQL + RDBMS
- GitLab CI/CD
- Kubernetes (온프레미스)
- 마이크로서비스 아키텍처
```

### 추론 가능한 것
```
- Python (FastAPI): 보안 솔루션 개발자 JD에 명시
- 데이터 크롤링/스캐닝: CTI 특성상 대량 데이터 수집 필요
- NoSQL: 대량 인텔리전스 데이터 저장 (Elasticsearch? MongoDB?)
- RDBMS: 서비스 메타데이터, 사용자 관리
- SaaS 형태: 웹 기반 서비스로 고객에게 제공
```

### DevOps 엔지니어가 담당할 것 (추론)
```
CTI 서비스 인프라:
  - 온프레미스 K8s 클러스터 구축/운영
  - AGATHA/ARTHUR/DASHIELL 서비스 배포
  - 대량 스캔 데이터 처리 파이프라인
  - 모니터링 + 장애 대응

CI/CD:
  - GitLab 기반 파이프라인 구축
  - 컨테이너화 + 배포 자동화

데이터 인프라:
  - 크롤링 결과 저장소 (NoSQL)
  - 검색/분석 엔진 (Elasticsearch 가능성)
  - 서비스 DB (RDBMS)

보안:
  - CTI 데이터 특성상 높은 보안 수준 요구
  - 인프라 자체의 보안 (접근 제어, 암호화, 감사 로그)
```

## 4. 채용 프로세스 상세

```
1. 서류 전형 + 포트폴리오 → ✅ 합격
2. 1차 인터뷰 (직무 리더 + CEO) → 준비 중
3. 처우 협의
4. 최종 합격
5. 3개월 수습(계약직) → 정규직 전환
```

**1차 인터뷰 특징**:
- 직무 리더 + CEO **동시 참석** → 기술 + 문화 fit 동시 평가
- 4명 팀이니 CEO가 직접 면접 → 비전/성장 의지 중요
- 스타트업 특성: "이 사람이 혼자서 인프라를 굴릴 수 있는가"가 핵심

## 5. 면접에서 보여줘야 할 것

### 5-1. "혼자서 인프라 전체를 할 수 있다"
```
내 증거:
- 홈랩 K8s: 5노드 클러스터를 혼자 설계→구축→운영→보안 220일+
- DevSecOps 파이프라인: 7단계 전체를 혼자 구현
- 관측성: PLG Stack + SLI/SLO + 알람 체계를 혼자 설계
- 트러블슈팅: 6건+ 직접 해결
```

### 5-2. "보안에 대한 이해가 있다"
```
CTI 회사의 DevOps → 일반 DevOps보다 보안 이해가 중요
내 증거:
- Defense in Depth 6계층 보안 아키텍처 직접 설계
- Falco IDS → Talon IPS 자동 격리 파이프라인
- SSH Brute Force 238K건 실제 탐지 + 대응
- kube-bench CIS Benchmark 47 PASS
- 넷코아텍 UTM IDS/IPS 실무 경험
```

### 5-3. "CTI 도메인에 관심이 있다"
```
면접 전 준비:
- MITRE ATT&CK PRE-ATT&CK 이해
- C2 서버 탐지 원리 학습
- 다크웹 인텔리전스 기본 개념
- AGATHA/ARTHUR 제품 동작 원리 이해
→ "CTI를 잘 안다"가 아니라 "학습하고 왔다"가 포인트
```

### 5-4. "스타트업에서 성장하고 싶다"
```
4명 팀 → "왜 대기업이 아니라 스타트업?"
답변 방향:
- "인프라 전체를 직접 설계하고 운영할 수 있는 환경을 원합니다"
- "홈랩에서 혼자 전체를 했던 경험이 스타트업 환경과 잘 맞습니다"
- "보안 도메인에서 성장하고 싶고, CTI 인프라를 직접 구축하고 싶습니다"
```
