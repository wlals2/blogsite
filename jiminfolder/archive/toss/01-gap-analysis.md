## 갭 상세 분석

### Gap 1: 로그 아키텍처 — "설계"가 아닌 "설치"에 머물러 있음

**토스가 원하는 것**:
```
"로그 아키텍처와 접근 제어 모델까지 만들어가며"
→ 어떤 로그를, 어디서, 어떤 수준으로, 왜 수집하는가?
→ 로그 간 상관 관계를 어떻게 분석하는가?
```

**현재 상태**:
```
✅ 있는 것:
  - Kubernetes Audit Log (Secret 추적, 노이즈 90% 제거)
  - Loki (Pod stdout 로그 수집)
  - Wazuh Agent (호스트 로그 수집)
  - Falco (syscall 이벤트 → Loki/Wazuh 전달)

❌ 없는 것:
  1. "왜 이 로그를 이 수준으로 기록하는가" 설계 문서
     - Audit Log Policy는 설계했지만, 전체 로그 아키텍처 관점의 문서가 없음
     - 어떤 로그 소스가 있고, 각각 왜 필요하고, 어떻게 연결되는지

  2. 로그 간 상관 분석 사례
     - Wazuh에 로그가 쌓이지만, "A 이벤트와 B 이벤트를 연결해서 공격을 파악"한 적 없음
     - Falco 알림 → Wazuh에서 상관 분석 → 공격 체인 식별 경험 없음

  3. 로그 보관 정책
     - Audit Log: 30일 × 10백업 × 100MB (설계됨)
     - Loki: 기본값 사용 (의도적 설계 없음)
     - Wazuh: 기본값 사용

  4. 로그 기반 알림 규칙 설계
     - Falco → Discord는 있지만
     - "이 로그 패턴이 나오면 이렇게 대응" 매핑이 없음
```

**왜 문제인가**:
```
토스 면접관: "로그 아키텍처를 설계해보셨다고 했는데,
             어떤 기준으로 로그 수준을 결정하셨나요?"
현재 답변:   "Audit Log에서 Secret은 RequestResponse,
             시스템 노이즈는 None으로..."
면접관:      "그건 Audit Log 하나의 정책이고,
             전체 로그 아키텍처는 어떻게 되어있나요?"
현재 답변:   "..." (전체 그림이 없음)
```

**해야 할 것**:
```
[ ] 전체 로그 아키텍처 설계 문서 작성
    - 로그 소스 목록 (Audit Log, Pod 로그, Falco, Wazuh, Node 로그)
    - 각 소스별: 왜 수집하는가 + 수집 수준 + 보관 기간 + 분석 방법
    - 로그 흐름도 (소스 → 수집기 → 저장소 → 분석 → 알림)

[ ] Wazuh 상관 분석 규칙 작성 + 실제 테스트
    - Falco 이벤트 + Audit Log + SSH 로그를 연결하는 상관 규칙
    - "A 발생 후 5분 내 B 발생 시 CRITICAL" 같은 복합 조건

[ ] 로그 기반 실제 분석 사례 1건 이상 만들기
    - 공격 시뮬레이션 → 여러 로그 소스에서 흔적 확인 → 체인 분석
```

### Gap 2: 탐지→분석→대응→재발방지 루프 미완결

**토스가 원하는 것**:
```
"보안 설정 점검과 로그 분석, 이슈 대응을 통해 현장 문제를 직접 다뤄요"
→ 탐지만이 아닌, 분석→대응→재발방지까지의 완결된 사이클
```

**현재 상태**:
```
SSH Brute Force 사례:
  탐지: ✅ Wazuh가 238,903건 탐지
  분석: △ "7개 IP에서 반복 시도" 정도. 공격 패턴/시간대/타겟 분석 미흡
  대응: ✅ fail2ban + SSH 키 전용 + ufw
  재발방지: △ ADR-005 기록했지만 "다시 비슷한 공격이 오면?" 모니터링 체계 미흡
  리포트: ❌ 실무 양식 보고서 없음

OWASP ZAP 사례:
  탐지: ✅ Baseline Scan (FAIL 0, WARN 18)
  분석: ✅ WARN 항목별 원인 분석
  대응: ✅ HTTP 보안 헤더 6종 + Error Disclosure 패치
  재발방지: △ 재진단 미실시 (패치 후 재스캔 안 함)
  리포트: △ 블로그 포스트는 있지만 실무 양식 아님

Falco 탐지:
  탐지: ✅ 5개 시나리오 396건
  분석: ❌ "396건이 뭔지" 개별 분석 없음
  대응: △ Talon dryRun 상태 (실제 격리 안 함)
  재발방지: ❌ 커스텀 규칙 추가 경험 없음
  리포트: ❌ 없음
```

**왜 문제인가**:
```
토스 면접관: "Falco로 396건을 탐지했다고 했는데,
             그 중 실제 위협은 몇 건이었나요?"
현재 답변:   "..." (개별 분석을 안 했으므로 답할 수 없음)

토스 면접관: "탐지 후 어떻게 대응했나요?"
현재 답변:   "Talon이 자동으로... (dryRun이라 실제로는 안 했습니다)"
```

**해야 할 것**:
```
[ ] Falco 396건 개별 분석
    - 어떤 규칙에서 발생? (상위 5개 규칙별 분류)
    - 오탐 vs 실제 위협 분류
    - 실제 위협이면: 어떤 공격이었는가?

[ ] 자기 환경 공격 → 탐지 → 분석 → 대응 완결 사이클 1건
    - 직접 공격 시도 (예: Pod 내부에서 shell 실행)
    - Falco 탐지 확인
    - Wazuh에서 상관 로그 확인
    - Talon 격리 동작 확인 (dryRun 해제 또는 수동 확인)
    - 대응 후 검증 (격리된 Pod가 정말 통신 불가한지)

[ ] ZAP 패치 후 재진단
    - HTTP 헤더 6종 적용 후 재스캔
    - Before: WARN 18 → After: WARN ? (개선 수치)

[ ] 진단 리포트 작성 (실무 양식)
    - 취약점명, CVSS 점수, 재현 경로, 영향도, 대응방안, 조치 결과
```

### Gap 3: 실제 "로그를 보고 문제를 찾은" 경험 부재 (가장 치명적)

**토스가 원하는 것**:
```
"로그 분석, 이슈 대응을 통해 현장에서 발생하는 문제를 직접 다뤄요"
→ 로그를 직접 읽고, 패턴을 찾고, 원인을 진단한 경험
```

**현재 상태**:
```
❌ 로그를 직접 분석해서 문제를 발견한 사례가 없음

현재 패턴:
  1. kubectl get pods → CrashLoopBackOff 보임
  2. kubectl logs → 에러 메시지 확인
  3. 구글/AI에 에러 메시지 검색 → 해결

이건 "로그 분석"이 아니라 "에러 메시지 검색"임

토스가 기대하는 "로그 분석":
  1. Wazuh/Loki에서 이상 패턴 발견 (로그 양 급증, 특정 패턴 반복)
  2. 시간대별 상관 분석 (이 시간에 A와 B가 동시에 발생)
  3. 원인 추정 → 검증 → 대응
  4. 알림 규칙 추가 (재발 시 자동 탐지)
```

**왜 가장 치명적인가**:
```
토스 Security Infrastructure 팀의 일상 업무:
  "보안 설정 점검과 로그 분석, 이슈 대응"

→ "로그 분석"이 JD 업무 설명에 직접 들어있음
→ 이 경험이 없으면 "이 사람이 와서 뭘 할 수 있지?"라는 의문

현재 상태로 면접 가면:
  Q: "로그 분석으로 문제를 발견한 경험이 있나요?"
  A: "Wazuh에서 SSH brute force를 발견했습니다"
  Q: "그 외에는?"
  A: "..." (없음)
```

**해야 할 것**:
```
[ ] Loki에서 로그 패턴 분석 경험 만들기
    - 특정 기간 Pod 로그에서 ERROR 패턴 추출
    - 시간대별 에러 발생 빈도 분석
    - 원인 추정 → 확인 → 조치

[ ] Wazuh에서 보안 이벤트 분석 경험 만들기
    - MITRE ATT&CK 대시보드에서 이벤트 분류
    - 특정 Rule ID 기준 이벤트 드릴다운
    - "이 이벤트가 왜 발생했는가" 분석

[ ] 공격 시뮬레이션 → 다중 로그 소스 상관 분석
    - 공격 시도 → Falco(syscall) + Audit Log(API) + Wazuh(호스트) 세 곳에서 흔적 확인
    - "하나의 공격이 여러 로그 소스에 어떻게 나타나는가" 분석
```

### Gap 4: 부하 테스트 / 장애 시뮬레이션 경험 없음

**토스가 원하는 것**:
```
"현장에서 발생하는 문제를 직접 다뤄요"
→ 장애 상황에서 로그를 보고 대응한 경험
```

**현재 상태**:
```
❌ 부하 테스트 한 번도 안 해봤음
❌ 의도적 장애 시뮬레이션 안 해봤음
❌ "트래픽 폭증 → 어떤 일이 일어나는가" 모름

220일 무중단이라고 했지만:
  → 트래픽이 매우 적어서 무중단인 것
  → 부하 상황에서도 버티는지 검증 안 함
```

**해야 할 것**:
```
[ ] k6 또는 wrk로 부하 테스트
    - 정상 트래픽 × 10배 → 어디가 먼저 터지는가?
    - HPA가 실제로 동작하는가?
    - 응답 시간 변화 (p50, p95, p99)

[ ] 장애 시뮬레이션 (Chaos Engineering 기초)
    - Pod 강제 삭제 → PDB 동작 확인
    - 워커 노드 셧다운 → 서비스 영향도 확인
    - MySQL Pod 삭제 → 복구 시간 측정

[ ] 부하 중 보안 이벤트 분석
    - 부하 테스트 중 Falco가 오탐하는가?
    - Wazuh 로그 양이 폭증할 때 분석 가능한가?
```

### Gap 5: Kyverno/Admission 정책의 "왜" 부족

**현재 상태**:
```
✅ 4개 ClusterPolicy 운영 중
  1. Non-root 실행 강제
  2. 특권 컨테이너 차단
  3. 리소스 제한 필수
  4. latest 태그 차단

❌ 이 정책들이 "왜" 필요한지 공격 시나리오와 연결되지 않음
```

**토스 면접관이 물어볼 것**:
```
Q: "Non-root를 강제하는 정책을 만들었다고 했는데,
    Non-root가 아니면 실제로 어떤 공격이 가능한가요?"
A: "컨테이너 탈출이..." (구체적 시나리오 설명 가능한가?)

Q: "latest 태그를 차단하는 이유는?"
A: "불변성..." (그런데 실제로 latest 때문에 문제가 발생한 적 있는가?)
```

**해야 할 것**:
```
[ ] 각 Kyverno 정책별 "이 정책 없으면 어떤 공격이 가능한가" 정리
    - Non-root: root 컨테이너에서 호스트 파일시스템 접근 시나리오
    - 특권 컨테이너: /dev 접근 → 호스트 탈출 시나리오
    - latest 태그: 동일 태그 다른 이미지 → 서플라이 체인 공격

[ ] 가능하면 1개는 실제로 시연
    - Non-root 미적용 Pod에서 호스트 파일 읽기 시도 → 성공
    - Kyverno 적용 후 → Pod 생성 차단 확인
```

### Gap 6: etcd 암호화 미구현 (CIS 1.2.27/28)

**현재 상태**:
```
kube-bench 47 PASS, 0 FAIL, 12 WARN
  → 1.2.27/28: etcd Secret 암호화 WARN

현재 Secret은 etcd에 Base64(평문)으로 저장됨
→ etcd 백업 파일 유출 시 모든 Secret 노출
```

**토스 면접관**:
```
Q: "Secret 관리는 어떻게 하시나요?"
A: "SealedSecret으로 Git에 암호화해서 저장합니다"
Q: "etcd에는요?"
A: "..." (Base64 = 평문이라는 걸 알고 있지만 미조치)
```

**해야 할 것**:
```
[ ] etcd Secret 암호화 구현 (EncryptionConfiguration)
    - aescbc 또는 aesgcm 프로바이더 설정
    - 기존 Secret 재암호화
    - kube-bench 재실행 → WARN → PASS 전환 확인
```

### Gap 7: 네트워크 정찰/스캔 경험 없음

**토스 JD**:
```
"네트워크, 서버, OS 등 인프라의 동작 원리를 설명할 수 있는 분"

**현재 상태**:
```
❌ Nmap으로 자기 환경 스캔한 적 없음
❌ 열린 포트 → 불필요한 서비스 식별 → 차단 경험 없음
❌ 네트워크 레벨 보안 점검 경험 없음
```

**해야 할 것**:
```
[ ] Nmap으로 자기 클러스터 스캔
    - 각 노드별 열린 포트 확인
    - 불필요한 포트 식별 → CiliumNetworkPolicy 또는 ufw로 차단
    - Before/After 비교 (열린 포트 수 변화)
```

### Gap 8: 인증서/TLS 관리 경험 부족

**현재 상태**:
```
✅ cert-manager 네임스페이스 있음
❌ 실제로 TLS 인증서를 관리한 경험이 정리되지 않음
❌ mTLS는 Istio가 자동으로 해줌 → 직접 인증서를 다뤄본 적 없음

**토스 보안팀에서 인증서 관리는 핵심 업무**:
```
- 내부 서비스 간 TLS
- 인증서 갱신 자동화
- 만료 모니터링
```

**해야 할 것**:
```
[ ] cert-manager로 인증서 발급 → 갱신 → 만료 모니터링 경험 정리
    (이미 구축되어 있다면 현황 파악 + 문서화)
```

### MUST (이것 없으면 서류 약함)

| # | 갭 | 작업 | 예상 시간 |

## 이력서 필수 포함 3가지와 매칭

### A. "직접 고민하고 시도해본 경험"
```
현재 쓸 수 있는 것:
  ✅ Defense in Depth 설계 (왜 6계층인가)
  ✅ PERMISSIVE mTLS 선택 (왜 STRICT 아닌가)
  ✅ Falco-Talon Phase 전략 (UTM 오탐 경험 → 설계)

채워야 할 것:
  → 로그 아키텍처 설계 (왜 이 로그를 이 수준으로)
  → 공격 → 탐지 → 분석 → 대응 사이클
```

### B. "궁금 → 고민 → 시도 → 배움"
```
현재 쓸 수 있는 것:
  ✅ MySQL Protocol Detection (궁금: 왜 연결 안 됨? → 근본 원인 2단계 분석)
  ✅ kube-bench false positive (궁금: 왜 FAIL? → version_mapping 문제 발견)
  ✅ Audit Log .bak 파일 문제 (시도: 8가지 재시작 → 배움: manifests 내 모든 파일 파싱)

채워야 할 것:
  → "보안" 도메인에서의 궁금→시도→배움 사례 (위 3개는 인프라 중심)
  → 공격 시뮬레이션에서 "예상과 다른 결과 → 원인 분석 → 배움" 사례
```

### C. "실패한 지점, 다시 하면 바꾸고 싶은 점"
```
현재 쓸 수 있는 것:
  ✅ Terraform State 삭제 사고 → DynamoDB Lock 도입
  ✅ JDBC + Istio "안 된다" 표면적 이해 → 근본 원인 분석으로 관점 전환

채워야 할 것:
  → 보안 관점에서의 실패 (예: "처음에 보안 없이 운영 → SSH BF 발견 → 그제서야 보안 구축")
  → "다시 하면?" (예: "처음부터 로그 아키텍처를 설계하고 시작했을 것")
```
