## Phase 1: 로그 분석 + 탐지 대응 — 1~2일

**목표**: "로그를 보고 문제를 찾은 경험" 1건 이상 확보
**이것이 가장 급한 이유**: 토스 JD 업무 설명에 "로그 분석, 이슈 대응"이 직접 들어있음

### 1-1. Falco 기존 396건 분석

```
작업:
  1. Falco 이벤트 조회 (Loki 또는 Wazuh에서)
  2. 규칙별 분류 (상위 5개 규칙)
  3. 오탐 vs 실제 위협 판별
  4. 판별 기준 + 결과 정리

산출물:
  - "Falco 396건 중 실위협 X건, 오탐 Y건 (오탐률 Z%)"
  - 이력서 한 줄: "Falco 이벤트 396건 분석, 오탐률 X% 산출 → 규칙 튜닝"

예상 시간: 1-2시간
```

### 1-2. 자기 환경 공격 → 다중 로그 소스 상관 분석

```
시나리오 (예시):
  1. Pod 내부에서 shell 실행 (kubectl exec)
  2. 민감 파일 읽기 시도 (/etc/shadow)
  3. 외부 통신 시도 (curl)

확인할 것:
  - Falco: syscall 이벤트 발생? 어떤 규칙?
  - Audit Log: kubectl exec API 호출 기록?
  - Wazuh: 호스트 레벨 로그?
  - Loki: Pod 로그?

산출물:
  - "하나의 공격이 4개 로그 소스에 어떻게 나타나는가" 분석 문서
  - 시간순 이벤트 체인 (타임라인)
  - 이력서 한 줄: "공격 시뮬레이션 → 4개 로그 소스 상관 분석 → 공격 체인 식별"

예상 시간: 2-3시간
```

### 1-3. Falco 탐지 → Talon 격리 → 검증 완결

```
작업:
  1. 공격 시뮬레이션 (1-2에서 이어서)
  2. Falco 탐지 확인
  3. Talon 동작 확인 (dryRun 해제 또는 수동 확인)
  4. 격리된 Pod가 실제로 통신 불가한지 검증
  5. 격리 후 복구 절차 실행

산출물:
  - "탐지(5초) → 격리(자동) → 통신 차단 확인 → 복구" 전체 사이클
  - 이력서 한 줄: "Falco IDS → Talon IPS 자동 격리 검증 (탐지→격리 5초, 통신 차단 확인)"

예상 시간: 1-2시간
```

## Phase 3: 로그 아키텍처 + 부하 테스트 — 1~2일

### 3-1. 로그 아키텍처 설계 문서

```
작업:
  전체 로그 흐름을 하나의 문서로 정리

내용:
  1. 로그 소스 목록
     | 소스 | 수집기 | 저장소 | 보관 | 왜? |
     |------|--------|--------|------|-----|
     | Pod stdout | Loki (DaemonSet) | Loki | 7일 | 애플리케이션 디버깅 |
     | Falco syscall | Falcosidekick | Loki + Wazuh | 30일 | 런타임 보안 |
     | Audit Log | kube-apiserver | 파일 | 30일 | API 접근 추적 |
     | Node syslog | Wazuh Agent | Wazuh Indexer | 90일 | OS 레벨 보안 |
     | Istio access | Envoy | Loki | 7일 | L7 트래픽 분석 |

  2. 로그 수준 결정 기준
     - Security-critical (Secret, exec, 삭제): 상세 기록
     - 시스템 노이즈 (health check, kube-proxy): 제거
     - 그 외: 메타데이터만

  3. 로그 흐름도 (다이어그램)
  4. 알림 규칙 매핑 (어떤 로그 → 어떤 알림)

산출물:
  - docs/에 로그 아키텍처 문서
  - 이력서 한 줄: "5개 로그 소스 × 4개 저장소 로그 아키텍처 설계"

예상 시간: 2시간
```

### 3-2. 부하 테스트

```
작업:
  1. k6 설치 + 테스트 스크립트 작성
  2. 정상 부하 → 10배 → 50배 단계별
  3. 측정: 응답 시간(p50/p95/p99), 에러율, HPA 동작
  4. Grafana에서 부하 중 메트릭 캡처

산출물:
  - "정상 p95=Xms → 10배 부하 p95=Yms → HPA Z→W replicas 확장"
  - 이력서 한 줄: "k6 부하 테스트 → SLO 기반 성능 검증 (p95 < 500ms)"

예상 시간: 2-3시간
```

### 3-3. 장애 시뮬레이션

```
작업:
  1. Pod 강제 삭제 → 서비스 영향도 + PDB 동작 확인
  2. 워커 노드 셧다운 → 서비스 복구 시간 측정
  3. MySQL Pod 삭제 → 데이터 무결성 + 복구 시간

산출물:
  - "워커 노드 다운 → 서비스 복구 X초 (PDB + TopologySpread)"
  - "MySQL Pod 삭제 → 자동 재생성 X초 + 데이터 무결성 확인"

예상 시간: 2시간
```

## Phase 4: 이력서 + 자소서 완성 — 1일

### 4-1. 이력서 보안 엔지니어 버전 완성

```
핵심 역량 5줄 (토스 맞춤):
  1. 컨테이너 보안 아키텍처 설계: 6계층 Defense in Depth
  2. 실제 위협 탐지·대응: SSH BF 238K건 + 공격 시뮬레이션 → 상관 분석
  3. 로그 아키텍처 설계: 5개 소스 × 4개 저장소, Audit Log 노이즈 90% 제거
  4. 보안 자동화: 7단계 CI/CD 파이프라인 + Falco-Talon Phase 전략
  5. 의사결정 기록: ADR 6건, 보안·생산성 균형 설계 (PERMISSIVE mTLS 등)
```

### 4-2. 자소서 작성 (토스 필수 3가지)

```
A. "직접 고민하고 시도해본 경험"
   → Defense in Depth 설계 과정 + 로그 아키텍처 설계
   → 링크: GitHub(k8s-manifests) + 블로그

B. "궁금 → 고민 → 시도 → 배움"
   → MySQL Protocol Detection 근본 원인 분석
   → 또는: 공격 시뮬레이션에서 예상과 다른 결과 → 원인 분석

C. "실패한 지점 + 다시 하면 바꾸고 싶은 점"
   → "처음에 보안 없이 운영 → SSH BF 발견 → 사후 보안 구축"
   → "다시 하면 처음부터 로그 아키텍처를 설계하고, 보안을 기본으로 시작했을 것"
```
