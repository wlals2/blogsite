## OpenTelemetry 연결 포인트 (★ 핵심 차별화)

### 내가 구현한 OTel 스택 전체

```
Spring Boot WAS (opentelemetry-javaagent.jar v1.32.0)
  │  ↓ OTLP gRPC (4317)
  │
Grafana Tempo (grafana/tempo:2.3.1)
  ├── OTLP gRPC/HTTP receiver (4317, 4318)
  ├── Jaeger receiver (14250, 14268)
  ├── metrics_generator
  │     ├── span-metrics (Trace → RED 메트릭 자동 생성)
  │     └── service-graphs (서비스 간 의존성 시각화)
  └── Prometheus remote write → Prometheus
           ↓
        Grafana 대시보드 (APM 패널)
```

### 이게 잔디 공고에서 말하는 OpenTelemetry와 어떻게 연결되는가

```
잔디 공고: "OpenTelemetry 기반 환경 경험"

→ OpenTelemetry = Traces + Metrics + Logs의 통합 관측성 표준

내가 구현한 것:
  Traces  → OTLP receiver로 Tempo에 수집, Grafana에서 시각화
  Metrics → span-metrics processor로 Trace → RED 메트릭 자동 생성 → Prometheus
  Logs    → Loki (별도)

코드 변경 없이 자동 계측 (Auto-instrumentation):
  → Java Agent를 initContainer로 주입 → JAVA_TOOL_OPTIONS 환경변수로 활성화
  → Spring Boot 코드 수정 없이 HTTP 요청/DB 쿼리/외부 호출 자동 추적

실제 운영 중:
  → blog.jiminhome.shop WAS의 API 호출 추적 (Spring Boot → MySQL)
  → Grafana Tempo 대시보드에서 TraceID로 개별 요청 드릴다운 가능
```

### 실제 사례: Tempo로 병목 발견 → 로직 삭제

```
[상황]
  WAS(Spring Boot)에 AI 뉴스를 외부에서 가져오는 로직이 있었음
  서비스를 운영하던 중 WAS 응답이 느려지는 현상 발생

[Tempo로 발견]
  Grafana Tempo에서 WAS 트레이스를 확인
  → AI 뉴스 수집 로직이 많은 HTTP 요청을 동시에 발생시켜
    대기열(Queue)이 쌓이고 있었음
  → 정상적인 사용자 요청까지 지연되는 것을 TraceID로 확인

[해결]
  해당 AI 뉴스 수집 로직 삭제
  → WAS 응답 정상화

[의미]
  로그만으로는 "느리다"만 알 수 있었음
  Tempo의 분산 트레이싱으로 "어느 로직이 왜 느리게 만드는가"를
  코드 수준까지 식별 가능 → 즉각적 해결
```

→ **"도구를 구축했다"가 아닌 "도구로 실제 문제를 발견하고 해결했다"**

### 면접에서 설명하는 방법

```
"Grafana Tempo로 분산 트레이싱을 운영하다가 실제 문제를 발견한 경험이 있습니다.
WAS 응답이 느려지는 현상이 있었는데, 로그로는 원인을 찾기 어려웠습니다.
Tempo에서 TraceID를 따라가 보니, AI 뉴스를 수집하는 로직이
대량의 외부 HTTP 요청 대기열을 만들어 정상 사용자 요청까지 지연시키고 있었습니다.
해당 로직을 삭제하고 정상화됐습니다.
로그로는 '느리다'밖에 몰랐는데, Trace로는 어디서 막히는지 바로 보였습니다."
```

### 3. 협업 (공통 — 오타 수정 필요)

```
멀티 클라우드 DR 프로젝트에서 팀장을 맡았습니다. 초반 DR 파트의 진행이 2주간 정체되면서
프로젝트 일정에 위기가 왔고, 이를 해결하기 위해 데일리 미팅을 하루 2회로 늘려 각 파트의
방향성과 진행 상황을 직접 점검했습니다.
그 과정에서 비용 효율(Cold Standby)과 복구 속도(Warm Standby) 사이에서 팀원 간 기술적
이견이 팽팽하게 대립했습니다.
저는 막연한 주장이 아닌 데이터로 소통했습니다. 직접 Terraform 스크립트로 PoC를 진행하여,
"Cold Standby 방식이라도 스크립트 최적화 시 RTO 30분 내 복구가 가능하며, 비용은 98%
절감된다"는 구체적인 수치를 제시했습니다.
이를 통해 팀원 모두가 납득하여 합의점을 도출했습니다. 이 과정에서 무엇보다 중요하게
여긴 것은 '동료의 의견에 대한 존중'이었습니다. Warm Standby 방식 역시 틀린 것이 아니라,
현재의 예산 제약상 우선순위가 다를 뿐임을 분명히 했습니다.
다른 의견도 '추후 고도화 과제'로 기록하여 팀원의 기술적 고민이 헛되지 않도록 배려하였습니다.
이후 의사 결정 속도가 빨라졌고, 기한 내에 프로젝트를 성공적으로 완수할 수 있었습니다.
```

### 4. 강점 (공통)

```
- 첫째, 새로운 기술은 직접 환경을 구축하며 익힙니다.
쿠버네티스를 처음 공부할 때, 문서만으로는 실무 감각을 익히기 어렵다고 판단했습니다.
PC 장비에 직접 클러스터를 구축하면서 FD 고갈, Cilium의 kube-proxy 대체 문제 등 수많은
장애를 겪었고 그 과정에서 리눅스 커널과 네트워크 설정을 몸으로 익혔습니다.
이후에도 Cilium(eBPF 기반 네트워킹)과 Falco(syscall 기반 런타임 감시)처럼 커널 레벨에서
동작하는 도구들을 직접 운영하며 시스템 내부 구조에 대한 이해를 넓혀가고 있습니다.

- 둘째, 적극적으로 묻고 공유하며 팀과 함께 성장합니다.
멀티 클라우드 프로젝트 당시, Karpenter 운영 문제로 실무 경험의 한계에 부딪혔습니다.
혼자 시간을 보내기보단, 즉시 AWS KRUG(AWS 한국 사용자 모임)에 참여해 현직 엔지니어
분들께 조언을 구했습니다. 덕분에 문제를 해결한 것은 물론, 알게된 노하우를 부트캠프
동료들에게 전파하는 'Learn Do Share'를 실천하여 그 공로로 표창장을 받았습니다.
팀 프로젝트에서도 기술적 이견이 있을 때 PoC 수치로 근거를 만들어 합의를 이끌어낸
경험이 있습니다. 입사 후에도 이런 지식 공유 문화를 이어가며 팀에 도움이 되는
엔지니어가 되겠습니다.

- 셋째, 배운 것을 기록하여 자산으로 남깁니다.
온프레미스 K8s를 운영하며 겪은 장애와 해결 과정을 기술 블로그에 기록하고 있습니다.
단순 메모가 아니라 각 기술 도입 배경과 트레이드오프 분석을 아키텍처 문서로 정리하여
같은 문제를 반복하지 않는 기반을 만들었습니다. 입사 후에도 선배님께 먼저 다가가 묻고,
배운 것은 기록하고 팀과 나누며 함께 성장하는 엔지니어가 되겠습니다.

서비스가 멈추지 않는 인프라를 만드는 것, 그것이 제가 엔지니어로서 지키고자 하는 가치입니다.
```

## 냉정한 매칭 분석

### 강점 (차별화)

```
1. OpenTelemetry 완전 구현 (우대사항, 신입 중 드묾)
   → OTel Java Agent + Tempo + OTLP + span metrics + Prometheus remote write
   → "경험 있다"가 아닌 "실제로 운영 중인 시스템을 보여줄 수 있다"

2. Kubernetes 실운영 (우대사항, 신입 중 가장 강한 카드)
   → 베어메탈(kubeadm 직접 설치) + EKS 양쪽
   → 220일 운영, 501회 배포 실패 0, 실제 장애 직접 해결

3. "기본기 갖춘 엔지니어" 자격요건에 정확히 부합
   → 최신 유행 도구가 아닌, kubeadm/kubelet/etcd 원리 이해
   → ADR로 기술 선택 이유를 문서화하는 습관
   → Linux 베어메탈 서버 직접 운영

4. B2B SaaS 특성 이해 + SLI/SLO 실수치
   → 99% 가용성 SLO, Canary 배포, 실시간 모니터링 체계
   → "측정하지 않은 것은 개선할 수 없다" 원칙으로 운영

5. 1인 풀스택 인프라 운영 경험
   → CI/CD + 모니터링 + 보안 + 네트워크 전부 직접 선택/구축/운영
   → 35명 소규모 팀에서 인프라 담당 역할에 적합
```

### 약점 (솔직하게)

```
1. 대용량 트래픽 처리 경험 없음
   → 홈랩과 Bespin 프로젝트 수준의 트래픽
   → 잔디는 40만 팀이 쓰는 서비스 → 스케일이 다름
   → 면접 답변: "대용량은 직접 경험 없으나, HPA/Karpenter 자동 스케일링 설계 및
                부하 시뮬레이션을 통한 성능 검증 경험이 있습니다"

2. Elastic Stack 운영 경험 없음 (잔디가 Elastic Stack 사용 중)
   → ELKF IaC 프로젝트 (Ansible로 4대 서버 프로비저닝)로 일부 커버
   → 면접 답변: "Loki(로그)와 Elasticsearch 설치 경험은 있으나 프로덕션 운영은 없습니다.
                학습 의지는 충분합니다."

3. Python/Go 스크립트 제한적
   → bash 스크립트 운영 중, Python 기초 수준
   → 우대사항이므로 감점 요인은 아님
```
