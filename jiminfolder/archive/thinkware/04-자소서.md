## 항목 1. 지원 동기

물류 현장 관리자로 근무하던 시절, PDA 단말기 접속 장애 한 번에 1만 5천 개의 물류가 멈추는 상황을 반복해서 목격했습니다. 수백만 원의 손실이 발생해도 현장에서 할 수 있는 일은 복구를 기다리는 것뿐이었습니다. 그때부터 "인프라를 직접 다루는 사람이 되고 싶다"고 생각했고, 이후 클라우드·인프라 엔지니어로 방향을 정했습니다.

팅크웨어를 선택한 이유는 세 가지입니다.

첫째, 아이나비는 블랙박스·내비게이션 브랜드파워 22년 연속 1위 기업입니다. 수십만 대의 차량에 연결된 커넥티드 서비스, BMW 80개국 OEM 공급, 자율주행 지도까지 — 단순 사내 IT가 아닌, 실제 서비스와 연결된 인프라를 운영할 수 있는 환경입니다. 물류 현장에서 배운 것처럼, 인프라는 비즈니스와 직결됩니다. 그 무게를 아는 환경에서 일하고 싶습니다.

둘째, IT 인프라 전 영역을 경험할 수 있는 구조입니다. 서버, 네트워크, 클라우드, 보안장비까지 한 팀에서 담당하는 환경은 인프라 엔지니어로서 기반을 다지기에 최적입니다. 클라우드만, K8s만 다루는 환경이 아니라 온프레미스부터 클라우드까지 전체를 이해하는 엔지니어가 되고 싶습니다.

셋째, 제가 가장 자신 있는 영역이 팅크웨어의 우대사항과 정확히 맞습니다. 보안솔루션 운영 경험과 온프레미스 인프라 구축 경험 — 이 두 가지는 홈랩 토이 프로젝트가 아닌, 실무와 실제 운영 환경에서 쌓은 것입니다. 그 경험을 팅크웨어에서 실제로 활용하고 싶습니다.

## 항목 2. 직무 적합성 — 제가 강한 이유 3가지

### 1. 보안솔루션 운영 경험 (우대사항 직접 충족)

넷코아텍에서 UTM 기반 IDS/IPS를 실무로 운영했습니다. 방화벽 정책 관리, DMZ 구성, IDS/IPS 룰 설정을 직접 담당했고, 가장 중요한 교훈을 현장에서 배웠습니다. **자동 차단 정책을 광범위하게 적용했을 때 정상 트래픽까지 차단되는 오탐 장애**가 반복됐습니다. 그 경험에서 "보안 자동화는 심각도에 따라 단계적으로 적용해야 한다"는 원칙을 체득했습니다.

이 원칙은 현재 운영 중인 온프레미스 K8s 환경의 보안 아키텍처 설계로 이어졌습니다.

- **Falco(IDS)**: 컨테이너 런타임 이상 행동을 커널 syscall 수준에서 탐지
- **Talon(IPS)**: 오탐 경험을 반영한 Phase 전략 — CRITICAL(즉시 격리) / WARNING(알림만)
- **Wazuh(SIEM)**: 전체 보안 이벤트 수집·분석·MITRE ATT&CK 매핑

실제 탐지 결과: SSH Brute Force 238,903건, 5개 공격 시나리오 396건, 웹 스캐너 17건.

UTM 장비 실무 운영 → 오탐 문제 경험 → 코드 기반 보안 자동화 설계. 이 흐름이 제가 보안솔루션을 단순히 "설치"가 아닌 "운영"할 수 있다는 근거입니다.

---

### 2. 온프레미스 인프라 전반 구축·운영 경험 (Private Cloud)

2025년 7월부터 현재까지 220일 이상, 베어메탈 서버에 Kubernetes 클러스터를 직접 구축하고 운영하고 있습니다. 클라우드 관리형 서비스(EKS)가 자동으로 해주는 것들 — Control Plane 관리, etcd 백업, kubelet 서비스 관리, 인증서 갱신 — 을 직접 담당합니다. 이것이 Private Cloud 운영 경험의 본질입니다.

구성 요소:
- **서버**: Ubuntu 22.04 베어메탈 5대 직접 구축 (Control Plane 1 + Worker 4)
- **스토리지**: Longhorn 분산 블록 스토리지 (4노드 분산, 노드 장애 시 데이터 보존)
- **네트워크**: Cilium eBPF L3/L4 정책, MetalLB LoadBalancer, Istio L7 트래픽 제어
- **자동화**: ArgoCD GitOps → 501회 배포, 실패 0건, 문제 발생 시 롤백 10초

Linux 서버 운영에서 실제 겪고 해결한 장애들:
- kubelet 중단 → Pod Terminating 멈춤 → SSH 진입 → systemctl 재시작
- Falco eBPF File Descriptor 고갈 → ulimit 조정
- Istio + MySQL Protocol Detection 교착 → 포트명 변경으로 해결

장애가 나도 혼자 진단하고 해결해야 하는 환경에서 220일을 버텼습니다. 팅크웨어 IT 인프라팀의 실제 업무 환경과 가장 가까운 경험이라고 생각합니다.

---

### 3. AWS + Terraform: 클라우드 인프라 설계 경험

멀티클라우드 DR 프로젝트(팀장, 4명)에서 AWS 전체 아키텍처를 직접 설계했습니다.

- VPC: Public/Private Subnet 분리, Multi-AZ (ap-northeast-2a+2c)
- EKS: 멀티AZ 클러스터, HPA + Karpenter 오토스케일링
- Route53: Health Check + Failover (87초 자동 전환)
- Terraform: 전체 인프라 코드화 (S3 State + DynamoDB Lock, 팀 협업 충돌 방지)
- 성과: RTO 6시간 목표 → **30분 달성**, 비용 추가 **거의 없음**

온프레미스(Private Cloud)와 클라우드(AWS) 양쪽을 직접 운영하면서 두 환경의 차이와 연결 방식을 이해하고 있습니다. 팅크웨어의 "Private / Public Cloud(AWS) 구축/관리" 업무를 양쪽에서 지원할 수 있습니다.

## 항목 3. 장애 대응 경험

현장 경험에서 배운 것은 하나입니다. **장애는 피할 수 없다. 하지만 빠른 진단과 복구는 설계할 수 있다.**

대표 사례 2건:

**① 네트워크 보안장비 오탐 장애 (넷코아텍)**

UTM IPS 자동 차단 정책 적용 후, 정상 업무 트래픽이 공격 시그니처와 유사하다는 이유로 차단됐습니다. 사용자들이 특정 외부 서비스에 접속 불가 상태가 됐고, 즉시 UTM 로그를 분석해 차단 IP와 패턴을 확인했습니다. 정상 트래픽임을 파악한 후 Whitelist에 추가하고, 이후 자동 차단 임계값을 재검토해 재발을 막았습니다. 이 경험에서 "보안 정책은 효과만큼 영향 범위도 설계해야 한다"는 것을 배웠습니다.

**② 서버 장애 → Pod 전면 중단 (홈랩)**

K8s 워커 노드의 kubelet 서비스가 중단되면서 해당 노드의 Pod들이 Terminating 상태로 수 시간째 복구되지 않았습니다. `kubectl get pods`만 보면 Pod 문제처럼 보이지만, `kubectl get nodes`로 내려가보니 해당 노드가 NotReady 상태였습니다. SSH로 직접 접속, `journalctl -u kubelet`으로 오류 로그를 확인하고 `systemctl restart kubelet`으로 복구했습니다. **증상이 아닌 계층을 따라가며 근본 원인을 찾는 것** — 이것이 제 장애 대응 원칙입니다.

## 항목 4. 입사 후 목표

솔직하게 말씀드리겠습니다. 저는 팅크웨어 IT 인프라팀 업무 중 **L2 네트워크 장비 실습 경험이 부족**합니다. IDC 현장 경험도 없습니다.

Windows Server는 부트캠프와 개인 학습 과정에서 VMware 환경을 구성하여 직접 다뤄본 경험이 있습니다. IIS 웹서버 기본 구성, DHCP 스코프 설정 및 IP 고갈(Starvation) 공격 실습, Active Directory OU 구성과 시간 기반 접근 정책 설정, NTFS/공유 권한 관리를 학습했습니다. 실제 운영 환경 경험은 없으나 구성과 동작 원리는 이해하고 있습니다.

그리고 Linux 베어메탈 서버 운영, 보안솔루션 실무 경험, Private/Public Cloud 구축 경험은 신입 중 드문 조합이라고 생각합니다. 없는 것은 빠르게 배우면 됩니다. 쿠버네티스를 공부할 때도 직접 베어메탈에 클러스터를 구축하면서 배웠고, AWS를 공부할 때도 실제 멀티클라우드 DR을 직접 설계하면서 배웠습니다.

입사 후 3개월 목표:
- Windows Server 실운영 환경 적응 (AD 그룹 정책, 서비스 관리, 도메인 운영)
- 팀의 네트워크 장비 구성 파악 및 L2/L3 실습
- 기존 인프라 구성을 문서화하고 이해하는 것에서 시작

팅크웨어의 IT 인프라는 블랙박스 커넥티드 서비스, 자율주행 지도, 유통 ERP까지 다양한 도메인이 얽혀 있습니다. 그 복잡한 환경을 직접 운영하면서, 인프라 엔지니어로서 깊이를 더하고 싶습니다.

서비스가 멈추지 않는 인프라를 만드는 것. 그것이 제가 이 직무를 선택한 이유입니다.
