
### Q5. "장애 대응 경험을 하나 설명해주세요"

**왜 물어보는가**: JD — "시스템 장애 발생 시 신속한 대응 및 문제 해결"

**답변 가이드 (2가지 준비)**:

**사례 1: kubelet 장애 (네트워크/서버 관점)**
```
"Worker 노드 kubelet이 중단되어 해당 노드의 Pod들이
Terminating 상태로 수 시간째 멈추는 장애가 있었습니다.

진단 과정:
1. kubectl get pods → Terminating 상태 확인
2. kubectl get nodes → 해당 노드 NotReady 발견
   (Pod 문제처럼 보여도 노드 레벨이 문제였음)
3. SSH로 해당 서버 직접 접속
4. systemctl status kubelet → inactive (dead) 확인
5. journalctl -u kubelet -n 50 → 오류 로그 확인

해결: sudo systemctl restart kubelet → 노드 Ready → Pod 정상화

배움: 장애 진단은 항상 '증상'이 아니라 '계층을 내려가며 근본 원인'을 찾아야 한다."
```

**사례 2: 넷코아텍 UTM 오탐 장애**
```
"UTM IPS 자동 차단 정책 적용 후 정상 트래픽까지 차단된 장애였습니다.

상황:
- 공격 시그니처와 유사한 패턴을 가진 정상 업무 트래픽이 차단됨
- 사용자들이 특정 외부 서비스에 접속 불가

진단:
- UTM 로그에서 차단 이벤트 확인
- 차단된 IP/패턴 분석 → 정상 비즈니스 트래픽임을 확인

해결:
- 해당 IP를 Whitelist에 추가
- 자동 차단 정책의 임계값 재검토

배움: 자동 차단은 오탐 위험이 있다.
이 경험에서 '심각도 높은 이벤트만 자동 차단, 나머지는 알림 후 수동 확인'
원칙을 체득하게 됐다."
```

### Q8. "모니터링 시스템 구축 경험이 있나요?"

**답변 가이드**:
```
"Prometheus + Grafana + Loki + Tempo (PLG Stack)를 직접 구축했습니다.

각 역할:
- Prometheus: 메트릭 수집 + PromQL로 SLI/SLO 계산
- Grafana: 대시보드 (3단계: SLO 개요 → 서비스 상세 → 분석)
- Loki: 로그 수집 (Falco 보안 이벤트 + Pod 로그 통합)
- Tempo: 분산 트레이싱 (병목 원인 코드 레벨까지 확인)

실제 성과:
- WAS 응답 지연 원인을 Tempo 트레이스로 발견 (AI 뉴스 수집 API 호출 병목)
- MySQL 백업 CronJob 실패를 Pushgateway 메트릭으로 감지 → Discord 알람
- Discord 실시간 알람 구성 (Grafana Alerting)

SLI/SLO 체계:
- Availability: Blackbox Exporter로 외부 엔드포인트 30초마다 probe
- Latency: p95 < 500ms 목표
- Error Rate: 5xx < 1% 목표"
```

### Q9. "네트워크 L3/L4 구성 경험은?"

**답변 가이드**:
```
"K8s 환경에서 Cilium eBPF를 사용해 L3/L4 네트워크 정책을 운영했습니다.

구체적으로:
- CiliumNetworkPolicy로 Pod 간 접근 제어
  예: 'WAS Pod만 MySQL 3306 포트에 접근 가능'
  → 다른 Pod에서 MySQL 접근 시도는 커널 레벨에서 차단
- eBPF는 iptables를 우회하므로 기존 방식보다 성능이 좋음
- Hubble UI로 네트워크 흐름 시각화

또한 MetalLB로 온프레미스에서 L4 LoadBalancer를 구현했습니다:
- Kubernetes Service type=LoadBalancer를 온프레미스에서 사용 가능하게
- externalTrafficPolicy=Local로 클라이언트 원본 IP 보존

AWS에서는 VPC/Subnet/Security Group/ALB까지 직접 설계한 경험이 있습니다."
```
