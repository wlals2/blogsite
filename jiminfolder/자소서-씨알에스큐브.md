## 질문 1. 회사에 지원이유가 무엇인가요? 지원 동기에 대해 말씀 부탁드립니다.

과거 물류 현장 관리자로 근무하며, PDA 단말기 접속 장애가 발생할 때마다 1만 5천 개의 물류가 멈추는 상황을 수없이 목격했습니다. 접속 장애 한 번에 수백만원의 물류 지연 비용이 발생해도, 현장에서는 복구를 기다리는 것 외에 할 수 있는 일이 없었습니다. 그때부터 "인프라를 직접 다루는 사람이 되고 싶다"고 생각했습니다.

CRScube는 20개국 3,500건 이상의 임상시험에 AWS 기반 SaaS 솔루션을 제공하며 99.99% 가용성을 유지하고 있습니다. 임상시험 데이터는 한번 손상되면 되돌릴 수 없고, 서비스 중단은 환자 안전과 직결됩니다. 물류 현장에서 "인프라가 멈추면 비즈니스가 멈춘다"는 것을 몸으로 배운 만큼, CRScube에서 그 서비스 연속성을 직접 책임지는 SRE가 되고 싶습니다.

또한 CRScube가 매년 25~30% 성장하며 북미 시장(메드넷 인수), 일본·중국 법인까지 확장하고 있다는 점에서, 글로벌 인프라를 안정적으로 운영하는 SRE의 역할이 앞으로 더 중요해질 것이라고 생각합니다. 성장하는 회사에서 인프라의 기반을 함께 만들어나가고 싶습니다.

## 질문 2. 본인이 지원 직무에 적합하다고 생각하시는 이유가 무엇인가요?

### 1. 컨테이너 기반 배포 자동화 경험

ECS를 직접 운영한 경험은 없지만, EKS(관리형)와 베어메탈 kubeadm(직접 구축) 양쪽을 운영하며 컨테이너 네트워킹, 스토리지, 배포 전략, 롤백 메커니즘의 원리를 깊이 이해하고 있습니다.

온프레미스 K8s 운영 시, 수동 배포의 위험성을 고려하여 GitOps 정책(수동 변경 금지)을 수립하고 ArgoCD를 도입했습니다. 나아가 Canary 배포(Argo Rollouts)를 적용하여 신규 버전의 트래픽을 단계적으로 전환하는 무중단 배포 파이프라인을 구현했습니다. 결과적으로 501회 배포, 실패 0, 문제 발생 시 롤백 10초의 결과를 냈습니다.

### 2. Terraform 기반 IaC 및 DR 설계 경험

멀티 클라우드 DR 프로젝트(팀장/4명)에서 Terraform S3 Backend + DynamoDB Lock으로 협업 가능한 IaC 환경을 구성했습니다. Cold Standby와 Warm Standby의 트레이드오프를 분석하고, Cold Standby 아키텍처로 비용 98% 절감과 RTO 28분을 동시에 달성했습니다. 장애는 피할 수 없지만 복구 속도는 설계할 수 있다는 것을 이 프로젝트에서 배웠습니다.

### 3. 모니터링 및 관측성 체계 구축 경험

온프레미스 K8s 프로젝트에서 Prometheus + Grafana + Loki 기반 모니터링 체계를 구축하고, OpenTelemetry + Tempo를 활용한 분산 트레이싱을 직접 구축하고 운영했습니다. 이 스택은 Pinpoint·ELK가 제공하는 것과 동일하게 메트릭·로그·트레이스를 모두 커버합니다. SLO 기반의 알람 체계를 설계하여, 장애가 터진 후가 아닌 이상 징후 단계에서 감지하는 체계도 직접 만들었습니다.

### 4. 보안 관제 및 네트워크 실무 경험

넷코아텍에서 11개월간 DMZ 구간 내 UTM 기반 경계 보안 아키텍처를 설계·구축한 경험이 있습니다. 방화벽 정책 최적화, 보안 존 세분화, IPS/IDS/Anti-Virus 맞춤형 정책 적용 등 네트워크 보안 실무를 수행했습니다. 이 경험은 AWS 환경에서 Security Group, WAF, 네트워크 ACL 등 보안 설정을 이해하는 기반이 되었습니다.

또한 온프레미스 K8s에서 Cilium(L3/L4) + Istio(L7) 계층별 네트워크 정책을 설계하고, Falco(행동 기반 IDS) + Wazuh(로그 기반 SIEM)를 결합한 보안 관제 체계도 직접 구축했습니다.

### 5. 직접 부딪히며 익히는 실행력

쿠버네티스를 처음 공부할 때, PC 장비에 직접 클러스터를 구축하면서 FD 고갈, Cilium의 kube-proxy 대체 문제 등 수많은 장애를 겪었고, 그 과정에서 리눅스 커널과 네트워크 설정을 몸으로 익혔습니다. 현재 220일간 무중단 운영을 이어가고 있으며, 겪은 장애와 해결 과정을 기술 블로그(blog.jiminhome.shop)에 기록하고 있습니다.

멀티 클라우드 프로젝트 당시에는 Karpenter 운영 문제로 한계에 부딪혔을 때, AWS KRUG(한국 사용자 모임)에 참여해 현직 엔지니어 분들께 조언을 구하고, 알게 된 노하우를 동료들에게 전파하여 표창장을 받았습니다.

서비스가 멈추지 않는 인프라를 만드는 것, 그것이 제가 엔지니어로서 지키고자 하는 가치입니다.

### 2. [목표 및 역량] "인프라는 곧 비즈니스다"라는 원칙으로 서비스 연속성에 기여하겠습니다.

물류 현장에서 배운 교훈과 프로젝트 경험을 바탕으로, 입사 후 다음 3가지를 목표로 삼겠습니다.

- 첫째, 컨테이너 기반 배포 자동화로 서비스 연속성을 보장하겠습니다.

- 둘째, Terraform으로 확장 가능한 AWS 인프라를 코드로 관리하겠습니다.

### 3. [협업 경험] 데이터 기반 소통과 상호 존중으로 팀의 난관을 돌파합니다.

멀티 클라우드 DR 프로젝트에서 팀장을 맡았습니다. 초반 DR 파트의 진행이 2주간 정체되면서 프로젝트 일정에 위기가 왔고, 이를 해결하기 위해 데일리 미팅을 하루 2회로 늘려 각 파트의 방향성과 진행 상황을 직접 점검했습니다.
그 과정에서 비용 효율(Cold Standby)과 복구 속도(Warm Standby) 사이에서 팀원 간 기술적 이견이 팽팽하게 대립했습니다.
저는 막연한 주장이 아닌 데이터로 소통했습니다. 직접 Terraform 스크립트로 PoC를 진행하여, "Cold Standby 방식이라도 스크립트 최적화 시 RTO 30분 내 복구가 가능하며, 비용은 98% 절감된다"는 구체적인 수치를 제시했습니다.
이를 통해 팀원 모두가 납득하여 합의점을 도출했습니다. 이 과정에서 무엇보다 중요하게 여긴 것은 '동료의 의견에 대한 존중'이었습니다. Warm Standby 방식 역시 틀린 것이 아니라, 현재의 예산 제약상 우선순위가 다를 뿐임을 분명히 했습니다.
다른 의견도 '추후 고도화 과제'로 기록하여 팀원의 기술적 고민이 헛되지 않도록 배려하였습니다. 이후 의사 결정 속도가 빨라졌고, 기한 내에 프로젝트를 성공적으로 완수할 수 있었습니다.

### 4. [강점] 직접 부딪히고 물어서 해결하는 실행력이 있습니다.

- 첫째, 새로운 기술은 직접 환경을 구축하며 익힙니다.
쿠버네티스를 처음 공부할 때, 문서만으로는 실무 감각을 익히기 어렵다고 판단했습니다. PC 장비에 직접 클러스터를 구축하면서 FD 고갈, Cilium의 kube-proxy 대체 문제 등 수 많은 장애를 겪었고 그 과정에서 리눅스 커널과 네트워크 설정을 몸으로 익혔습니다. 덕분에 새로운 환경이나 도구를 접해도 빠르게 적응하고 활용할 수 있는 기반이 되었습니다.

- 둘째, 적극적으로 묻고 공유하며 성장합니다.
멀티 클라우드 프로젝트 당시, Karpenter 운영 문제로 실무 경험의 한계에 부딪혔습니다. 혼자 시간을 보내기 보단, 즉시 AWS KRUG(AWS 한국 사용자 모임)에 참여해 현직 엔지니어 분들께 조언을 구했습니다. 덕분에 문제를 해결한 것은 물론, 알게된 노하우를 부트 캠프 동료들에게 전파하는 'Learn Do Share'를 실천하여 그 공로로 표창장을 받았습니다.

- 셋째, 배운 것을 기록하여 자산으로 남깁니다.
온프레미스 K8s를 운영하며 겪은 장애와 해결 과정을 기술 블로그에 기록하고 있습니다. 단순 메모가 아니라 각 기술 도입 배경과 트레이드 오프 분석을 아키텍처 문서로 정리하여 같은 문제를 반복하지 않는 기반을 만들었습니다. 입사 후에도 선배님께 먼저 다가가 묻고, 배운 것은 기록하고 팀과 나누며 함께 성장하는 엔지니어가 되겠습니다.

서비스가 멈추지 않는 인프라를 만드는 것, 그것이 제가 엔지니어로서 지키고자 하는 가치입니다.
