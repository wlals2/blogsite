## 0. 가장 큰 문제: 하나의 이력서로 모든 포지션에 지원하고 있다

### 현재 상태
```
클라우드 엔지니어, 보안 엔지니어, SRE 전부 같은 이력서로 지원
→ 핵심 역량 5줄이 각각 다른 방향을 가리킴
→ 자소서 목표 3개가 각각 다른 직무
→ 면접관 30초 스캔 결과: "이 사람 뭘 하고 싶은 거지?"
```

### 해결: 이력서 2개 버전 + 포트폴리오 2개 버전
```
버전 A: 클라우드 엔지니어
  → 인프라 설계/운영이 메인, 보안은 부가 역량
  → DR, GitOps, 오토스케일링, IaC가 앞에

버전 B: 보안 엔지니어
  → 보안 설계/자동화가 메인, 인프라는 부가 역량
  → UTM→Falco+Talon, Phase 전략, DevSecOps가 앞에
```

## 2. 넷코아텍 UTM 스토리 (수정됨 - 사실 기반)

### 이전 버전 (잘못된 표현)
```
❌ "탐지는 되는데 차단은 사람이 해야 한다"
→ 사실이 아님. UTM에도 자동 차단 기능은 있었음
```

### 수정된 버전 (실제 경험)
```
✅ "UTM의 자동 차단 기능은 있었지만, 오탐률이 높아
정상 트래픽까지 차단되는 문제가 반복됨.
결국 자동 차단을 신뢰할 수 없어 수동으로 정책을 관리해야 했음."
```

### Falco+Talon으로 연결되는 스토리
```
넷코아텍 UTM (오탐 문제로 수동 관리)
  → 이 경험이 Phase 전략의 설계 근거
  → CRITICAL만 즉시 격리, WARNING은 1주 관찰
  → "오탐으로 정상 서비스가 죽는 걸 직접 겪었기 때문에 나온 설계"
```

### 자소서에 넣을 문장 (보안 엔지니어 버전)
```
"넷코아텍에서 UTM 기반 IDS/IPS를 운영하며,
자동 차단 기능의 오탐률이 높아 정상 트래픽까지 차단되는 문제를 겪었습니다.
결국 수동으로 정책을 관리해야 했고,
'자동화하려면 오탐 리스크를 반드시 설계에 반영해야 한다'는 교훈을 얻었습니다.

이 경험을 바탕으로 Falco+Talon 체계에서는
CRITICAL(쉘 접속, 민감 파일 변조)만 즉시 격리하고,
WARNING은 라벨만 붙여 1주일 관찰 후 판단하는 Phase 전략을 설계했습니다."
```

## 3. 이력서 버전 A: 클라우드 엔지니어

### 컨셉
```
메인 무기 = 인프라 설계 + 안정적 운영
보조 무기 = 보안 (한 줄)
메시지 = "안정적인 인프라를 설계하고 운영합니다"
```

### 핵심 역량 (5줄 → 방향 통일)
```
1. 멀티 클라우드 DR 설계 (RTO 28분, 비용 98% 절감)
2. EKS + Karpenter 오토스케일링으로 트래픽 유연 대응
3. ArgoCD 기반 GitOps + Canary 무중단 배포 파이프라인 구축
4. Terraform IaC로 멀티 클라우드 프로비저닝 자동화
5. 베어메탈 K8s 220일 무중단 운영 (모니터링 + 알람 체계 직접 구축)
```
→ 전부 "인프라 설계/운영" 방향으로 통일

### 자소서 흐름
```
1. [직무 지원 계기] 물류 현장에서 인프라의 무게를 배웠습니다 (현재 유지)
2. [목표] 안정적 인프라 설계와 운영으로 기여하겠습니다
   → 첫째: DR/HA 아키텍처 (RTO 28분 달성 경험)
   → 둘째: GitOps 무중단 배포 (개발팀 지원)
   → 셋째: 모니터링 + 알람으로 장애 5분 내 파악
   → (보안은 "운영 중 보안도 설계의 일부로 다뤘습니다" 한 줄)
3. [협업] 데이터 기반 소통 (현재 유지 - 좋음)
4. [강점] 실행력 (현재 유지)
```

### 프로젝트 순서
```
1순위: 멀티 클라우드 DR (팀장, 규모, 비용 최적화)
2순위: 베어메탈 K8s (운영 안정성, 모니터링)
→ 보안(Falco, Cilium)은 2순위 안에서 한 줄로
```

## 4. 이력서 버전 B: 보안 엔지니어

### 컨셉
```
메인 무기 = 보안 운영 경험 + 자동화
보조 무기 = 인프라 (한 줄)
메시지 = "현장 경험을 바탕으로 보안을 자동화합니다"
```

### 핵심 역량 (5줄 → 방향 통일)
```
1. 넷코아텍 UTM 기반 IDS/IPS 운영 (오탐 대응 정책 설계)
2. DevSecOps 파이프라인: GitLeaks → Trivy → Falco → Talon
3. Falco-Talon Phase 전략 설계 (오탐 리스크 고려한 단계별 격리)
4. Cilium(L3/L4) + Istio(L7) 계층별 Zero Trust 구현
5. Wazuh SIEM 연동으로 로그 기반 상관 분석 체계 구축
```
→ 전부 "보안 설계/자동화" 방향으로 통일

### 자소서 흐름
```
1. [직무 지원 계기]
   → 넷코아텍 UTM 운영에서 오탐 문제 경험
   → "자동화하려면 오탐을 설계에 반영해야 한다" 교훈
   → 이것이 보안 엔지니어의 길로 이끔

2. [목표] 보안 위협을 자동으로 탐지하고 대응하는 체계를 만들겠습니다
   → 첫째: DevSecOps 파이프라인 (빌드~런타임 전구간)
   → 둘째: Phase 전략 (오탐 겪었기에 나온 설계)
   → 셋째: SIEM + 모니터링 연계 (Wazuh + Falco + Grafana)
   → (인프라는 "보안이 적용된 K8s 클러스터를 직접 구축/운영합니다" 한 줄)

3. [협업] 데이터 기반 소통 (현재 유지)
4. [강점] 실행력 (현재 유지)
```

### 프로젝트 순서
```
1순위: Zero Trust Bare Metal K8s (보안 설계, DevSecOps)
2순위: 멀티 클라우드 DR (클라우드 보안 아키텍처)
→ DR은 2순위 안에서 "보안 관점"만 강조
```

## 9. 면접 질문 준비 (확인된 약점)

### Wazuh / Falco / 보안 아키텍처

| 질문 | 정확한 답변 |
|---|---|
| Wazuh가 뭔가요? | 로그 기반 상관 분석으로 보안 이벤트 탐지 및 알람 SIEM |
| Falco와 차이? | Wazuh: 로그 기반 / Falco: syscall 행동 기반, Zero-day 가능 |
| 격리하면 서비스 중단? | NetworkPolicy로 트래픽만 차단, Replica 3개로 서비스 지속 |
| Phase 전략이 뭔가요? | 오탐 방지 위해 CRITICAL 즉시 격리, WARNING은 1주 관찰 |
| UTM과 Falco 차이? | UTM: 규칙 기반(오탐 많음), Falco: 행동 기반(syscall, 더 정밀) |
| WARNING 1주 관찰 중 공격자가 움직이면? | 라벨 붙인 후 Grafana MITRE ATT&CK 대시보드에서 집중 모니터링. 단, 자동 에스컬레이션 조건(N회 반복 시 CRITICAL 격상)은 미구현 → 개선 포인트로 인정 |

### 아키텍처 심화 (Q6~Q12: 날카로운 질문)

| 질문 | 핵심 포인트 | 100점 답변 요소 |
|---|---|---|
| **Q6** PERMISSIVE mTLS인데 왜 STRICT 안 써? | MySQL sidecar 미주입 때문 | MySQL은 sidecar 없어 STRICT 시 인증 불가 → 격리 전략으로 보완 (mTLS는 Pod간, MySQL은 Service 격리) |
| **Q7** MySQL sidecar 왜 금지? | Envoy가 MySQL 바이너리 프로토콜을 HTTP처럼 중간에서 끊음 | TCP 프록시는 가능하지만 L7 파싱 불가 → connection 불안정. 공식 권고도 DB에 sidecar 비권장 |
| **Q8** Falco 한계: syscall 우회 시? | eBPF/kernel module 우회 가능 | container_escape, kernel exploit으로 Falco 자체 무력화 가능. 보완: Wazuh 로그(후행), immutable 이미지, 최소 권한 |
| **Q9** GitLeaks 어디서 동작? Pre-commit vs CI | 현재: CI (push 시점) | Pre-commit은 개발자 로컬 필요. CI는 강제성 있음. 약점: push 이후 탐지 → 히스토리에 남음 → git filter-repo로 제거해야 |
| **Q10** ArgoCD selfHeal + 긴급 장애 시 직접 수정? | selfHeal=true면 kubectl 수정이 즉시 롤백됨 | 긴급 시: ArgoCD App 일시 suspend → kubectl 수정 → 원인 파악 → Git 반영 → unsuspend |
| **Q11** Wazuh Rule 직접 작성? | Falco→Wazuh 연동 rule 작성 | custom rule XML 작성. group=falco_alerts, level=12 (CRITICAL). 직접 작성 경험 있음 |
| **Q12** DR RTO 28분 어떻게 검증? | 실제 failover 테스트 | curl -X POST + HTTP 200 확인. Route53 TTL 60초 감안 DNS 전파 대기. 28분 = DNS(2분) + EKS 기동(15분) + 서비스 헬스체크(11분) |
