## 냉정한 매칭 분석 (면접관 관점)

### 강점 (차별화)
```
1. K8s 베어메탈 + EKS 양쪽 경험 (신입 중 드묾)
   → kubeadm으로 직접 구축 → 내부 동작 이해 → EKS에서 "왜 이렇게 동작하는지" 설명 가능

2. 220일 실제 운영 + 501회 배포 실패 0
   → 테스트 환경이 아닌 실제 서비스 운영 증거
   → blog.jiminhome.shop (현재도 운영 중)

3. GitOps 전체 파이프라인 (공고 기술스택 100% 일치)
   → GitHub Actions → ArgoCD → Argo Rollouts Canary
   → 이것이 공고에서 요구하는 "CI/CD 파이프라인 구축"의 직접 증거

4. Observability 실수치 보유
   → SLI/SLO 체계, CPU Requests 22.4→6.25 Core (-72%)
   → 단순 구축이 아닌 "운영 효율화" 증거

5. DevSecOps 파이프라인
   → GitLeaks → Trivy → Kyverno → Falco+Talon+Wazuh
   → 추가 역량으로 어필 가능 (공고 요구는 아니지만 플러스)
```

### Ansible 경험 (있음 — 수준 "중-하")
```
ELKF IaC 프로젝트: https://github.com/wlals2/elkf-IaC
  - site.yml → 5개 playbook 오케스트레이션 (bootstrap → ES → LS → KB → Filebeat)
  - inventory.ini → 4대 서버 멀티노드 구성 (ES/LS/KB/Web)
  - group_vars/all.yml → 변수 중앙 관리 (버전, URL, 포트)
  - Logstash 파서: Dissect → Grok → Ruby 3단계 폴백 전략
  - handler 체인: validate → restart → health check

수준:
  - playbook/inventory/group_vars 구조는 이해함
  - AI 도움을 받아 작성했고, 처음부터 코드를 짜본 건 아님
  - 면접에서: "ELKF 스택 4대 서버를 Ansible로 프로비저닝한 경험이 있습니다.
    playbook 구조, inventory, 변수 관리는 이해하고 있으나,
    복잡한 role을 처음부터 짜본 경험은 아직 부족합니다." (솔직하게)

이력서 반영:
  - 스킬 태그에 "Ansible" 추가 가능
  - 프로젝트 설명에 "Ansible로 ELKF 멀티노드 프로비저닝" 한 줄 추가
```

### 약점 (솔직하게)
```
1. 블록체인 노드 운영 경험 없음
   → 우대사항이므로 없어도 감점 요인 아님
   → 기술적 관심은 진정성 있게 표현 가능
   → "블록체인 노드도 결국 서버 운영" — K8s 운영 경험 이전 가능성 강조
```

### 1. 직무 지원 계기 (알로카도스 특화 버전)

```
물류 현장에서 PDA 장애로 현장이 멈추는 것을 보며 인프라의 무게를 배웠습니다.

물류 현장에서 재고 입력에 쓰던 PDA 시스템이 다운되는 순간을 겪었습니다.
현장 작업자 수십 명이 멈추고, 관리자가 수작업으로 기록하는 혼란을 보며
"인프라가 멈추면 사람도 멈춘다"는 것을 체감했습니다.

그 경험이 인프라 엔지니어로의 방향을 결정했고, 이후 AWS EKS 기반 3-Tier 아키텍처에서
팀장을 맡아 RTO 28분의 DR 아키텍처를 설계했으며, 현재는 베어메탈 K8s 클러스터를
220일 이상 실제 서비스로 운영하며 파이프라인 전체를 직접 구현하고 있습니다.

알로카도스에 지원하는 이유는 두 가지입니다.

첫째, 기술 스택이 제가 지금까지 구축한 것과 정확히 일치합니다. AWS, Kubernetes,
GitHub Actions, ArgoCD, Prometheus, Grafana — 단순히 사용해본 수준이 아니라,
직접 설계하고 220일 이상 운영하며 트러블슈팅까지 해본 기술들입니다.

둘째, 블록체인 인프라에 대한 관심입니다. 블록체인 노드는 결국 서버입니다.
24시간 가동, 네트워크 분리, 슬래싱 리스크 대응, 노드 장애 시 복구 자동화 —
이것은 제가 지금 하고 있는 일(고가용성, 자동 복구, 모니터링)과 다르지 않습니다.
Web 3.0 인프라를 운영하며 새로운 도메인의 경험을 쌓고 싶습니다.
```

**자체 평가**: 93점
- 강점: PDA 에피소드 → 기술 스택 일치 → 블록체인 관심까지 자연스럽게 연결
- 약점: 블록체인 파트가 약간 억지스러울 수 있음 → 솔직하게 "노드 운영 경험 없음"을 인정하고 K8s 경험 연결로 보완
- 글자수: 약 580자

### 2. 목표 및 역량 (기존 Version A 그대로 사용)

> 아래는 기존 클라우드 엔지니어 버전 자소서. 알로카도스에 그대로 사용 가능.

```
저는 '서비스가 멈추지 않는 인프라'를 세 가지 축으로 만든다고 생각합니다.
배포가 안전해야 하고, 장애에서 빠르게 복구해야 하며, 문제를 미리 감지할 수 있어야 합니다.

첫째, 배포 안정성입니다.
ArgoCD 기반 GitOps와 Argo Rollouts Canary 배포를 직접 구축해 운영하고 있습니다.
현재 베어메탈 K8s 클러스터에서 501회 배포를 수행했고 배포 실패는 0건입니다.
배포가 코드로 관리되니 롤백도 git revert 한 줄로 가능합니다.

둘째, 장애 복구력입니다.
Bespin 프로젝트에서 AWS → Azure 멀티 클라우드 DR 아키텍처를 팀장으로 설계했습니다.
Cold Standby 방식으로 추가 비용 없이 RTO 28분을 달성했습니다.
K8s 홈랩에서도 kubelet 중단, Protocol Detection Deadlock 등 실제 장애를 직접 해결했습니다.

셋째, 사전 감지입니다.
Prometheus + Grafana + Loki 기반 PLG Stack을 직접 구축하고, SLI/SLO 체계를 정의해
운영하고 있습니다. Bespin 프로젝트에서는 9개 대시보드, 31개 알람을 통해 백업 CronJob 실패와
Redis CPU throttling을 배포 전에 선감지한 경험이 있습니다.

알로카도스의 인프라도 이 세 가지 축 위에서 안정적으로 운영하겠습니다.
```

**자체 평가**: 90점 (기존과 동일)
- 알로카도스에도 그대로 적용 가능. 수정 불필요.
