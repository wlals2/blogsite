
## 2. Kubernetes

> 근거: 베어메탈 홈랩 (1인, 220일+ 운영) + EKS (Bespin 프로젝트)

| 기술 | 수준 | 근거 |
|------|------|------|
| **kubeadm 클러스터 구축** | **상** | CP 1 + Worker 3 직접 구축, 업그레이드, 노드 추가/제거 경험 |
| **EKS 클러스터 설계** | **중-상** | Multi-AZ 설계, Karpenter/HPA 구성, Spot → On-demand 전환 판단 |
| **Deployment/StatefulSet** | **상** | 220일 운영, 리소스 최적화 (CPU -72%), 트러블슈팅 |
| **HPA** | **중-상** | CPU 70% + Custom Metrics (네트워크 100KB/s) |
| **Karpenter** | **중** | 노드 오토프로비저닝, Spot 삭제 → On-demand 전환 경험 (EKS) |
| **CronJob** | **중** | MySQL 백업 + S3 업로드 + Pushgateway 메트릭 |
| **PDB** | **중** | minAvailable: 1 설정 |
| **TopologySpreadConstraints** | **중** | AZ 분산 + ScheduleAnyway (EKS), 노드 분산 (홈랩) |
| **VPA** | **중-하** | Off 모드로 권장값 수집만 (실제 적용 X) |
| **RBAC** | **중-하** | ServiceAccount + ClusterRoleBinding 수준 |
| **리소스 관리** | **상** | 정책 기반 (실제 × 2배), 22.4 → 6.25 Core 최적화, Phase별 롤아웃 |

### K8s 트러블슈팅 경험
```
1. Istio + MySQL Protocol Detection Deadlock (Server-First 교착 → sidecar 제외)
2. kubelet 중단 → Pod Terminating 무한 대기 (노드 확인 → restart)
3. Falco File Descriptor Limit (eBPF FD 초과 → ulimit 조정)
4. ArgoCD + HPA 충돌 (ignoreDifferences 설정)
5. HTTPS Redirect Loop (X-Forwarded-Proto 미확인)
6. CloudFront Origin Group POST 차단 (Dual Domain 전략)
```

### K8s 종합 평가
```
수준: 상
이유: 베어메탈 직접 구축 (관리형이 해주는 것을 직접 해봄)
      + EKS 설계 (관리형에서 효율적으로 사용하는 법도 앎)
      + 220일 실운영 + 실제 트러블슈팅 6건+
```

### 의사결정 기록
```

- Istio PERMISSIVE 유지 → MySQL TCP 연결 지원 (ADR-002)
- MySQL sidecar 제외 → Server-First 프로토콜 교착 방지 (ADR-003)
- Nginx Ingress 제거 → Istio Gateway가 L7 라우팅 통합 (hop 2→1)
```

## 7. 관측성 (Observability)

| 기술 | 수준 | 근거 |
|------|------|------|
| **Prometheus** | **상** | PromQL 작성, ServiceMonitor, Recording Rules, SLI/SLO 체계 정의 |
| **Grafana** | **상** | 대시보드 직접 설계, Alerting (Discord), 3-Level 구조 (SLO → Detail → 분석) |
| **Loki** | **중** | 로그 수집/검색, LogQL 기본 |
| **Tempo** | **중** | 분산 트레이싱 구축 (OTEL Java Agent → Tempo → metrics_generator), 실제 병목 발견 경험 |
| **Alloy** | **중-하** | 메트릭 수집 DaemonSet |
| **Blackbox Exporter** | **중-하** | 외부 엔드포인트 가용성 모니터링 |
| **Pushgateway** | **중** | CronJob 메트릭 수집, 백업 실패 감지 |

### 관측성 실적
```
홈랩:
  - SLI/SLO 체계: Availability 99%, Latency p95 < 500ms, Error Rate 5xx < 1%
  - 대시보드 3-Level 구조
  - Discord 알람

Bespin:
  - 9개 대시보드, 31개 알람 (3-Tier: Critical/High/Medium)
  - 실제 발견: 백업 CronJob 실패 (Pushgateway), Redis CPU throttling (Discord)
```

### Tempo 실제 사용 사례

```
[상황]
  WAS(Spring Boot)에 AI 뉴스를 외부 API에서 수집하는 로직이 있었음
  서비스 운영 중 WAS 응답이 느려지는 현상 발생

[발견]
  Grafana Tempo에서 WAS 트레이스 확인
  → AI 뉴스 수집 로직이 외부 HTTP 요청을 다수 동시 발생 → 대기열(Queue) 누적
  → 정상 사용자 요청까지 지연되는 것을 TraceID로 확인

[해결]
  해당 AI 뉴스 수집 로직 삭제 → WAS 응답 정상화

[배움]
  로그(Loki) = "느리다" 만 알 수 있음
  트레이스(Tempo) = "어느 코드가 왜 느리게 만드는가" 를 코드 수준까지 식별 가능
```

### Tempo 구현 상세 (면접 설명용)

```
아키텍처:
  Spring Boot WAS
    → opentelemetry-javaagent.jar v1.32.0 (initContainer로 주입)
    → OTLP gRPC (4317) → Grafana Tempo

Tempo 설정:
  - OTLP gRPC/HTTP receiver (4317, 4318)
  - Jaeger receiver (14250, 14268)
  - metrics_generator:
      span-metrics   → Trace 데이터에서 RED 메트릭 자동 생성
      service-graphs → 서비스 간 의존성 시각화
  - Prometheus remote write → Prometheus에 메트릭 push

WAS 환경 변수:
  JAVA_TOOL_OPTIONS: "-javaagent:/otel/opentelemetry-javaagent.jar"
  OTEL_SERVICE_NAME: "board-was"
  OTEL_EXPORTER_OTLP_ENDPOINT: "http://tempo.monitoring.svc.cluster.local:4317"
  OTEL_TRACES_SAMPLER: "always_on"   (100% 샘플링)
  OTEL_METRICS_EXPORTER: "none"      (메트릭은 Prometheus 담당)
  OTEL_LOGS_EXPORTER: "none"         (로그는 Loki 담당)

핵심 포인트:
  - 코드 변경 없이 자동 계측 (Auto-instrumentation)
  - Spring Boot HTTP 요청 / DB 쿼리 / 외부 호출 자동 추적
  - Grafana에서 Trace ↔ Metrics 연결 (TraceID → 개별 요청 드릴다운)
```
